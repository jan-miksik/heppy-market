## Heppy Market – Targeted Code Review

_Date: 2026-03-10_

This document summarizes an objective review of the current codebase with a focus on **realistic security, correctness, and reliability risks**. The system is an **internal-only Cloudflare Worker API** behind a Cloudflare Pages service binding, providing **paper-trading agents and managers** on Base DEXes (no real funds flow).

Scope was limited to:

- Backend Worker (`apps/api/src/**`)
- Frontend Nuxt app (`apps/web/**`)
- Security-related docs in `docs/`

Severity levels below are calibrated to this context (internal-only API, paper trading) and avoid artificial dramatization.

---

## 1. Multi-tenant trade data exposure

- **Area**: API – trades route  
- **Files**: `apps/api/src/routes/trades.ts`  
- **Severity**: **Medium** (privacy / cross-tenant data), **not** financial loss  

### Description

`GET /api/trades` returns **all trades across all agents**:

- The route is protected by session auth (`createAuthMiddleware`), so only authenticated users can call it.
- However, the query does **not** scope by the authenticated user’s `walletAddress` or agent ownership:
  - It simply selects from `trades` (optionally filtered by `status` and `pair`) and returns up to 500 rows.
  - There is no join or filter tying each trade’s `agentId` back to an `agents.ownerAddress`.

In contrast, other APIs enforce ownership:

- `/api/agents/:id/*` and `/api/managers/:id/*` use `requireOwnership` / `requireManagerOwnership` helpers to ensure the caller owns the resource.
- Manual trade-closing (`POST /api/trades/:id/close`) verifies the agent is owned by the caller before allowing the action.

### Realistic impact

Given the current design:

- **Any authenticated user** can see trades generated by **other users’ agents**, including:
  - Pairs traded
  - PnL, amounts, and strategies
  - Timestamps and statuses
- There is **no direct financial risk** (trades are paper, not on-chain), but there is a **real privacy / competitive intelligence risk**:
  - A user can infer other users’ strategies and performance.
  - In a production multi-tenant SaaS, this would violate reasonable expectations of tenant isolation.

### Likelihood

- High: this is the default behavior of the route for every authenticated user.

### Recommendation

- Scope `/api/trades` by the authenticated user:
  - Join `trades` to `agents` and filter by `agents.ownerAddress == walletAddress` (similar to the ownership checks elsewhere).
  - Alternatively, restrict the endpoint to an admin role (if/when such a role exists) and hide it from normal users.
- For aggregate stats (`/api/trades/stats`), decide explicitly:
  - Either keep it **global** but document that users see anonymized cross-system stats, or
  - Add a `userScoped=true` mode that aggregates only the caller’s trades.

## 2. Full LLM prompts and responses logged to Worker console

- **Area**: LLM routing and trading logic  
- **Files**: `apps/api/src/services/llm-router.ts`  
- **Severity**: **Low–Medium** (privacy / log volume), depends on production logging policy  

### Description

`getTradeDecision` logs:

- The **entire system + user prompt** (`fullPrompt`), including:
  - Portfolio state
  - Market data
  - Behavior configuration and persona markdown (`personaMd`)
- The **raw LLM text response** for every model tried.

These logs are written with `console.log`/`console.warn`, which in Cloudflare Workers typically go to centralized logs where operators may search and store them long-term.

### Realistic impact

- The prompts and responses may contain:
  - User-provided persona text, which could include usernames, descriptions, or incidental PII (if users write it that way).
  - Strategy descriptions and risk parameters.
- There are **no direct secrets** (API keys, wallet private keys, etc.) logged here.
- Main risks:
  - **Privacy**: internal logs may reveal user-specific strategy text and personas to operators with log access.
  - **Volume**: prompts and responses can be large, increasing log size and making diagnostics noisy.

### Likelihood

- High in any environment where this code runs with logging enabled; every decision route will emit these logs.

### Recommendation

- For production:
  - Gate detailed prompt/response logging behind an environment flag (e.g. `LOG_LLM_DEBUG`) or a log-level check.
  - When disabled, log only high-level metadata (model id, latency, success/failure) and omit full prompt/response bodies.
- For debugging/dev:
  - Keeping this logging is useful; just ensure it is clearly documented as **debug-only** behavior.

## 3. Global, IP-based rate limiting only (no per-user limits)

- **Area**: API – global middleware  
- **Files**:
  - `apps/api/src/index.ts` – registration via `app.use('/api/*', createRateLimitMiddleware(...))`
  - `apps/api/src/lib/rate-limit.ts` – implementation  
- **Severity**: **Low** for this internal, paper-trading context  

### Description

The API applies a **single fixed-window rate limit per IP**:

- Limit: **60 requests / 60 seconds per IP** across all `/api/*` routes.
- Implementation notes:
  - Uses KV as a backing store (`rl:ip:${ip}:${window}` keys).
  - KV operations are non-atomic by design, so the limiter is **best-effort**, not strict.

There is **no per-user / per-wallet limit** and no additional endpoint-specific throttling (e.g. for high-cost routes like analysis runs).

### Realistic impact

Given the API is:

- Internal-only behind a Cloudflare Pages service binding, and
- Used for **paper trading** (no direct financial loss path),

the current limiter is likely **sufficient to prevent accidental overload** from a single client but:

- It does not prevent:
  - A motivated user from opening many agents and triggering many analysis runs.
  - Multiple clients behind the same IP (e.g. corporate NAT) from hitting shared limits or from one user affecting another.
- Under concurrency, some bursts may pass due to KV race conditions; this is an expected limitation of KV-based counters.

### Likelihood

- Moderate: any heavy UI polling or misconfigured client can hit the 60/60 window; abuse via multiple IPs is possible but would require intent.

### Recommendation

- If you later harden the platform:
  - Consider **per-user or per-wallet limits** for expensive operations (e.g. analysis triggers, manager decisions).
  - Keep the current IP-based limiter as a coarse first line of defense, but treat it as best-effort only (as already documented in comments).

## 4. Manual analysis and DO triggers can bypass DB “running” status

- **Area**: Durable Objects and routes controlling agents and managers  
- **Files**:
  - `apps/api/src/routes/agents.ts` – `/api/agents/:id/analyze`  
  - `apps/api/src/agents/trading-agent.ts` – `/analyze` handler  
  - `apps/api/src/routes/managers.ts` – `/api/managers/:id/trigger`  
  - `apps/api/src/agents/agent-manager.ts` – `/trigger` handler  
- **Severity**: **Low** (in paper-trading context), mostly about operator expectations  

### Description

There are manual endpoints that instruct Durable Objects to perform work immediately:

- Agent analyze:
  - API route checks ownership correctly (`requireOwnership`).
  - DO `/analyze` lazily initializes the engine if necessary and calls `runAgentLoop(..., { forceRun: true })`, then persists state and reschedules alarms if the agent is marked `running`.
  - Comment explicitly notes: “forceRun=true allows analysis even if agent is stopped/paused in DB”.
- Manager trigger:
  - `/api/managers/:id/trigger` checks ownership and requires DO status `running`, then schedules an alarm in 1 second.

This creates a gap between DB `status` and actual behavior:

- An agent marked `stopped` in the DB can still be **analyzed once** via the manual `/analyze` route.
- This is by design to allow manual “one-shot” analysis, but it may surprise operators who use the DB `status` as the single source of truth.

### Realistic impact

- For paper trading:
  - No direct financial risk—position opening and closing are still governed by the DO logic and available balance.
- Operationally:
  - A user could still run analysis on an agent they believe is “stopped” in the UI if the UI exposes the manual analyze action.
  - Could cause confusion if monitoring dashboards rely solely on DB `status` and ignore DO-level activity.

### Likelihood

- Moderate if the UI provides manual “Analyze now” buttons; low otherwise.

### Recommendation

- Ensure **UI semantics** are clear:
  - If an agent is “stopped”, either hide or clearly label manual analysis as a one-off dry run.
- If you want stricter behavior:
  - Make `/api/agents/:id/analyze` respect DB status (or require a specific “debug” flag), and pass `forceRun: false` in normal operation.

## 5. Logging of external API schema mismatches and raw key sets

- **Area**: Market data services  
- **Files**:
  - `apps/api/src/services/gecko-terminal.ts`  
  - `apps/api/src/services/dex-data.ts`  
- **Severity**: **Low** (diagnostic logs only)  

### Description

The external market data clients:

- Use **Zod schemas** to validate responses.
- On schema mismatch or unexpected shapes, they:
  - Log key sets and error messages, and
  - Throw structured errors.

Example:

- `gecko-terminal.ts` logs `raw response keys` before schema validation.

### Realistic impact

- No secrets or user-specific data are logged here; only external API responses.
- Logs could become noisy if the upstream API changes frequently, but the impact is mainly on **observability quality**, not security.

### Recommendation

- For production, consider:
  - Downgrading some of these logs to a debug trace level if they become noisy.
  - Keeping the current behavior is reasonable for now given the importance of catching upstream changes early.

## 6. General strengths observed

These are not risks but important context that **reduces** overall risk:

- **Internal-only Worker**:
  - The API Worker is not directly exposed; it is reachable only via the Cloudflare Pages service binding and the Pages proxy (`apps/web/server/api/[...path].ts`).
- **Authentication & session management**:
  - SIWE is verified with `viem`, with one-time nonces stored in KV, and nonces are **deleted after use**, preventing replay.
  - Sessions are stored in KV with explicit TTL and an `expiresAt` check; expired sessions are cleaned up.
  - Routes are clearly split into public (`/api/health`, `/api/pairs`, `/api/auth/*`) and protected (`/api/agents`, `/api/trades`, `/api/managers`, `/api/profiles`, `/api/compare`), with auth middleware applied by path prefix.
- **Authorization checks for core resources**:
  - Agent and manager routes use ownership helpers that enforce `ownerAddress == walletAddress`, with a deliberate “legacy unowned” allowance only where explicitly intended.
  - Manual trade closing also validates agent ownership.
- **Input validation and schema usage**:
  - Zod schemas are used systematically for request bodies and queries (`validateBody`, `validateQuery`), with structured `ValidationError` handling.
  - External API responses (DexScreener, GeckoTerminal) are validated with Zod before use, reducing the chance of runtime shape errors.
- **Durable Object safety patterns**:
  - Both agent and manager DOs have **auto-heal** logic for overdue alarms.
  - Alarm rescheduling is done in `finally` blocks so that errors in analysis or persistence do not leave agents/managers stuck in a “running with no next alarm” state.
  - Paper engine state is serialized/deserialized cleanly, with consistent use of timestamps and IDs.

---

## Summary

- The codebase shows **good discipline** around authentication, input validation, and Durable Object lifecycle management.
- The **most concrete risk** for a multi-tenant deployment is the **unscoped `/api/trades` listing**, which exposes all trades to any authenticated user; this should be addressed before real users rely on privacy between accounts.
- Other findings are mainly about **logging detail, rate limiting granularity, and behavioral semantics** (manual analysis vs. DB status), which are lower severity in this paper-trading, internal-API context but worth tightening if you evolve toward a production SaaS offering or real-money trading.
